## 🧪 Task A：自己生成数据，完成“可验证的推测”
这一部分不是让你随便造点正态分布数据，而是要你**自己定义一个明确的数据生成机制 (Data Generating Process, DGP)**，然后检查你的推断与设定是否一致。

### A1. 生成一份带业务含义的模拟回归数据
请你自己设计一个“看起来像真实业务”的数据生成场景。例如：
- 广告预算与销售额；
- 学习时长与考试成绩；
- 医疗投入与康复指标；
- 房屋特征与房价。

最低要求：
1. 样本量不少于 `300`；
2. 至少 `4` 个特征，其中：
   - 至少 `2` 个连续变量；
   - 至少 `1` 个类别变量；
   - 必须**明确构造一组高度相关的特征**，例如先生成 `x1`，再令 `x2 = 0.8 * x1 + 随机扰动`，从而让 `x1` 和 `x2` 在统计上明显相关；
3. 目标变量 `y` 必须由你显式设定生成公式；
4. 在数据中**主动加入真实世界问题**，至少包含其中两类：
   - 缺失值；
   - 异常值；
   - 特征量纲差异明显；
   - 共线性。

### A2. 保存数据并记录 DGP
你必须把生成后的数据保存到：
- `src/week11/data/synthetic_regression.csv`

并在 `synthetic_report.md` 中明确写出：
- 你设定的生成机制；
- 哪些变量应该正向影响 `y`；
- 哪些变量应该负向影响 `y`；
- 你是如何构造那组高度相关特征的；
- 哪些变量是冗余或高度相关的。

### A3. 用你的 `utils/` 流程完成建模与评估
你必须在模拟数据上完成以下流程：
1. 清洗数据（缺失值、异常值、编码等）；
2. 使用你自己的 `utils/transformers.py` 中的组件完成预处理；
3. 使用你自己的模型（如 `GradientDescentOLS` 或 `CustomOLS`）训练；
4. 使用 5 折交叉验证，且必须避免数据泄露；
5. 计算并报告至少三个指标：`RMSE`、`MAE`、`MAPE`；
6. 计算 VIF，并指出是否存在严重共线性问题。

### A4. 完成“推测”
这里的“推测”不只是报一个分数，而是回答：
- 你的模型识别出来的变量方向，是否与 DGP 一致？
- 如果不一致，是因为噪声、共线性，还是预处理问题？
- 哪些变量在你设计的数据里本来就难以稳定识别？为什么？

## 🛠️ 推荐工程结构（A）
为了帮助你把 `week11` 做成一条更清晰、可复用、可答辩的流程，建议你不要把所有逻辑都堆进一个巨大的 `main.py` 文件里，而是至少在 `main.py` 内部按函数拆分出几个明确阶段。

### 推荐的主流程函数划分
你可以参考下面这样的组织方式：
1. `generate_synthetic_data()`
   - 负责生成模拟数据并保存到 `data/`；
2. `run_synthetic_task()`
   - 负责读取模拟数据、完成清洗、诊断、无泄露 CV 和报告生成；
3. `load_kaggle_data()`
   - 负责读取你下载的 Kaggle 数据，并做最基本的字段检查；
4. `run_kaggle_task()`
   - 负责在真实数据上完成清洗、诊断、训练、评估和报告生成；
5. `write_report()`
   - 负责把关键结果写入 `results/*.md`；
6. `main()`
   - 作为唯一执行入口，依次调度以上函数。

### 推荐的主流程思路
建议你的 `main()` 至少体现出下面这种顺序：
- 先准备或清空 `results/`；
- 再生成并分析模拟数据；
- 再读取并分析 Kaggle 数据；
- 最后输出三份报告。

### 为什么这样拆分
这样做的好处有三点：
1. **更容易调试**：如果模拟数据部分有问题，不会和 Kaggle 数据流程混在一起；
2. **更容易复用**：你以后想单独重跑某个任务，只需要调用对应函数；
3. **更容易答辩**：你可以清楚地向老师解释“这一段代码是干什么的，那一段代码负责什么输出”。

### 推荐你尽量避免的写法
请尽量避免以下情况：
- 所有逻辑都堆在一个 300 行以上的脚本里；
- 数据读取、清洗、训练、评估、写报告交错在一起；
- 同样的预处理逻辑在模拟数据和 Kaggle 数据部分各复制一遍；
- `utils/` 已经有的能力，没有复用，而是临时再写一份。

一句话：
> Week 11 不只是完成两个任务，而是要把两个任务组织成一条你自己能讲清楚的工程流程。
